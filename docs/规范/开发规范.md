# 开发规范

## 1. 分支

- `master`只用于稳定主线。
- 新开发从 `master`创建 `codex/`前缀分支。
- 未明确要求时不提交、推送或合并。
- 合并完成后删除已合并的本地和远端开发分支。

## 2. Python

- 所有模块使用类型注解。
- 文件和路径统一使用 `pathlib.Path`。
- 数据结构优先使用 Pydantic模型或 `dataclass`。
- API层不直接操作SQLite、TWAIN或文件系统。
- 每个文件只承担一个明确职责。
- 不使用裸 `except:`。
- 不吞掉TWAIN、文件和数据库异常。

## 3. 测试驱动开发

每项功能执行：

```text
先写失败测试
→ 运行并确认失败原因正确
→ 实现最小代码
→ 运行并确认通过
→ 重构
→ 运行完整相关测试
```

普通单元测试禁止依赖真实扫描仪；设备测试使用明确 integration标记。

## 4. SQLite

- 只有主进程写数据库。
- 使用事务维护任务、页面和文件的一致性。
- 数据库迁移必须版本化。
- 禁止在SQLite中存储原图和缩略图二进制。

## 5. 多进程

- 入口必须调用 `multiprocessing.freeze_support()`。
- 命令和事件使用明确模型，禁止传递不可序列化对象。
- 工作子进程不依赖FastAPI对象。
- 工作进程退出必须由主进程回收并记录退出码。

## 6. 文件

- 先写临时文件，再原子重命名。
- 删除前校验规范化路径位于任务目录。
- 失败和中断文件不得成为有效页面。
- 原始JPEG不得被Pillow重新保存。

## 7. 日志

- 日志使用中文说明和稳定错误码。
- 日志按大小轮转。
- 不记录原图内容、密码、Token和无关本机文件。
- Capability大数组和二进制数据只记录摘要。

## 8. 完成验证

至少运行：

```powershell
pytest
ruff check .
python -m compileall app tests
git diff --check
```

涉及真实扫描仪时，还必须提供实机测试输出，不能用模拟测试替代。
